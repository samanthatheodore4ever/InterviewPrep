🎯 Key Tips from 1P3A

Distilled from candidate reports

  • 🔴 Speed is paramount — must finish ALL sub-problems with working, tested code. Incomplete = fail.
  • 🔴 Prepare ALL SD topics — the question pool is known. Prepare every single one.
  • 🟡 75-min "system coding" round — unique to OpenAI. Requires high-level design AND runnable code in one session.
  • 🟡 Infection spread is THE most common coding question — prepare all 5 variants thoroughly.
  • 🟡 Social Network with Snapshots is rising in frequency — OOD-focused.
  • 🟢 AI Safety — read OpenAI's AI Safety blog posts. Prepare personal examples.
  • 🟢 No level negotiation pre-interview — level determined by HC after all rounds.
  • 🟢 Results come fast — usually 1-3 days after onsite.
  • 🟢 Recruiter gives minimal info — don't expect topic hints. Prepare everything.
  • 🟢 Recruiter coding prompt (May 2026): "60 minutes, CoderPad, live coding. Data-structure heavy — then iterate as new constraints get introduced. We value steady progress, clear thinking, and strong communication."
  • 🟢 Tools: CoderPad for coding, Excalidraw for system design diagrams.
  • 🟡 HM fit matters as much as technical performance — "答得完美也不一定让你过，也许hm没看上" (perfect answers don't guarantee passing if HM doesn't see a fit).
  • 🟡 Design Sora is rising fast — appearing in both phone screens and onsites. It's fundamentally a task scheduler / resource allocation problem.

📋 Interview Format

Based on candidate reports from 1P3A

Standard SDE Process

  • Phone Screen: 2 back-to-back sessions (1h coding + 1h system design)
  • Onsite (VO): 5 rounds — 2× System Design, 1× Coding, 1× Technical Deep Dive, 1× HM Behavioral
  • Coding rounds: 60-75 min, detailed prompt, must produce working tested code
  • No fixed level targeting: Level (L4/L5/L6) determined at HC review after interviews
  • 75-min "system coding" round: Unique to OpenAI — requires BOTH high-level design AND runnable code
  • 15 min AGI/Safety discussion may be included in behavioral round
⚠️ Speed is everything. Multiple candidates report that incomplete code = instant fail. You must finish ALL sub-problems with working tests. Budget your time carefully.

Infection Spread Simulation
🔥 VERY HIGH
General Coding 60 min · 5 sub-problems
Matrix of plants: "X" (infected), "." (healthy), "I" (immune).
  • Q1: Infected plants spread to 8 neighbors each day. Days to equilibrium? → BFS
  • Q2: Add "I" immune cells that can't be infected. Same question. → Modified BFS
  • Q3: Infected plants recover after D days → become immune. Track infection day per cell with hashmap
  • Q4: Plants die if they ever have ≥K infected neighbors. Count deaths after D days
  • Q5: You can burn one entire row or column on any day. Minimize total deaths → Optimization/greedy
  • Variant: A cell only gets infected if it has ≥T infected neighbors
  • Variant: "Minimum Time to Infect All Plants" — return the number of days for infection to reach every cell (multi-source BFS shortest-time)
Sources: thread-1164914, thread-1171470, thread-1173190, thread-1178391, thread-1177818, thread-1178260, thread-1178236

Toy Language Type System
🔥 HIGH
General Coding 60 min
Implement a type system for a toy language:
  • Primitives: char, int, float
  • Generics: T1, T2, etc.
  • Tuples: [int, T1, char], can be nested [int, char, [int, T1]]
  • Functions: [int; [int, T1]; T2] -> [char, T2, [float, float]]

  • P2: Implement infer_return(function, params) — substitute generics, raise errors on type mismatch
